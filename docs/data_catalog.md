# Data Catalog for Gold Layer

## Overview

The Gold Layer provides a business-friendly, analytics-ready representation of the data warehouse.

It is structured as a dimensional model containing:

- Dimension tables describing customers and products
- A fact table containing sales transactions
- Surrogate keys used to link facts to dimensions
- Business-friendly column names and standardised values

The model is designed to support reporting, aggregation, filtering, and analytical queries.

---

## Gold Layer Model

```text
gold.dim_customers  1 ───────< gold.fact_sales >─────── 1  gold.dim_products
                      customer_key       product_key
```

---

## 1. `gold.dim_customers`

### Purpose

Stores one record per customer, combining CRM customer details with demographic and geographic information from the ERP source system.

### Grain

One row per customer.

### Primary Key

`customer_key`

### Columns

| Column Name       | Data Type     | Description                                                                                                   |
| ----------------- | ------------- | ------------------------------------------------------------------------------------------------------------- |
| `customer_key`    | `BIGINT`      | Surrogate key generated by the data warehouse to uniquely identify each customer dimension record.            |
| `customer_id`     | `INT`         | Original numerical customer identifier from the CRM source system.                                            |
| `customer_number` | `VARCHAR(50)` | Business-facing alphanumeric customer identifier used to match and reference customers across source systems. |
| `first_name`      | `VARCHAR(50)` | Customer's first name.                                                                                        |
| `last_name`       | `VARCHAR(50)` | Customer's surname or family name.                                                                            |
| `country`         | `VARCHAR(50)` | Standardised country of residence for the customer.                                                           |
| `marital_status`  | `VARCHAR(50)` | Standardised marital status of the customer, such as `Married`, `Single`, or `n/a`.                           |
| `gender`          | `VARCHAR(50)` | Standardised gender value, such as `Male`, `Female`, or `n/a`.                                                |
| `birthdate`       | `DATE`        | Customer's date of birth.                                                                                     |
| `create_date`     | `DATE`        | Date when the customer record was created in the CRM source system.                                           |

### Source Mapping

| Gold Column       | Primary Source                                                                                                         |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `customer_key`    | Generated in the Gold Layer using `ROW_NUMBER()`                                                                       |
| `customer_id`     | `silver.crm_cust_info.cst_id`                                                                                          |
| `customer_number` | `silver.crm_cust_info.cst_key`                                                                                         |
| `first_name`      | `silver.crm_cust_info.cst_firstname`                                                                                   |
| `last_name`       | `silver.crm_cust_info.cst_lastname`                                                                                    |
| `country`         | `silver.erp_loc_a101.cntry`                                                                                            |
| `marital_status`  | `silver.crm_cust_info.cst_marital_status`                                                                              |
| `gender`          | `silver.crm_cust_info.cst_gndr`, with `silver.erp_cust_az12.gen` used as a fallback when the CRM value is unavailable. |
| `birthdate`       | `silver.erp_cust_az12.bdate`                                                                                           |
| `create_date`     | `silver.crm_cust_info.cst_create_date`                                                                                 |

## 2. `gold.dim_products`

### Purpose

Stores product details enriched with category, subcategory, product-line, cost, and maintenance information.

### Grain

One row per current product.

### Primary Key

`product_key`

### Columns

| Column Name      | Data Type     | Description                                                                                       |
| ---------------- | ------------- | ------------------------------------------------------------------------------------------------- |
| `product_key`    | `BIGINT`      | Surrogate key generated by the data warehouse to uniquely identify each product dimension record. |
| `product_id`     | `INT`         | Original numerical product identifier from the CRM source system.                                 |
| `product_number` | `VARCHAR(50)` | Business-facing alphanumeric product identifier used to reference the product.                    |
| `product_name`   | `VARCHAR(50)` | Descriptive name of the product.                                                                  |
| `category_id`    | `VARCHAR(50)` | Identifier used to associate the product with its category information.                           |
| `category`       | `VARCHAR(50)` | High-level product classification, such as `Bikes`, `Components`, or `Clothing`.                  |
| `subcategory`    | `VARCHAR(50)` | More detailed classification within the product category.                                         |
| `maintenance`    | `VARCHAR(50)` | Indicates whether the product requires maintenance, such as `Yes` or `No`.                        |
| `cost`           | `INT`         | Product cost recorded by the source system, expressed in whole currency units.                    |
| `product_line`   | `VARCHAR(50)` | Standardised product line, such as `Mountain`, `Road`, `Touring`, or `Other Sales`.               |
| `start_date`     | `DATE`        | Date when the product record became effective or available for sale.                              |

### Current-Record Rule

Only current product records are included in the dimension.

For this project, a current product record is typically identified by:

```text
silver.crm_prd_info.prd_end_dt IS NULL
```

### Source Mapping

| Gold Column      | Primary Source                                   |
| ---------------- | ------------------------------------------------ |
| `product_key`    | Generated in the Gold Layer using `ROW_NUMBER()` |
| `product_id`     | `silver.crm_prd_info.prd_id`                     |
| `product_number` | `silver.crm_prd_info.prd_key`                    |
| `product_name`   | `silver.crm_prd_info.prd_nm`                     |
| `category_id`    | `silver.crm_prd_info.cat_id`                     |
| `category`       | `silver.erp_px_cat_g1v2.cat`                     |
| `subcategory`    | `silver.erp_px_cat_g1v2.subcat`                  |
| `maintenance`    | `silver.erp_px_cat_g1v2.maintenance`             |
| `cost`           | `silver.crm_prd_info.prd_cost`                   |
| `product_line`   | `silver.crm_prd_info.prd_line`                   |
| `start_date`     | `silver.crm_prd_info.prd_start_dt`               |

## 3. `gold.fact_sales`

### Purpose

Stores sales transaction line items and their associated measures for analytical and reporting use cases.

### Grain

One row per sales-order line item.

Each line item represents one product on a sales order. An order may therefore contain multiple rows.

### Dimension Relationships

| Foreign Key    | Referenced Dimension              |
| -------------- | --------------------------------- |
| `customer_key` | `gold.dim_customers.customer_key` |
| `product_key`  | `gold.dim_products.product_key`   |

### Columns

| Column Name     | Data Type     | Description                                                                                                                          |
| --------------- | ------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `order_number`  | `VARCHAR(50)` | Business identifier for the sales order. The same order number may appear on multiple rows when an order contains multiple products. |
| `product_key`   | `BIGINT`      | Surrogate foreign key linking the sales line item to `gold.dim_products`.                                                            |
| `customer_key`  | `BIGINT`      | Surrogate foreign key linking the sales line item to `gold.dim_customers`.                                                           |
| `order_date`    | `DATE`        | Date when the customer placed the order.                                                                                             |
| `shipping_date` | `DATE`        | Date when the order was shipped.                                                                                                     |
| `due_date`      | `DATE`        | Date by which the order was expected to be completed or delivered.                                                                   |
| `sales_amount`  | `INT`         | Total sales value of the line item, usually calculated as `quantity × price`.                                                        |
| `quantity`      | `INT`         | Number of units sold for the product on the order line.                                                                              |
| `price`         | `INT`         | Sales price per unit for the product on the order line.                                                                              |

### Measures

| Measure        | Aggregation        | Description                                                 |
| -------------- | ------------------ | ----------------------------------------------------------- |
| `sales_amount` | `SUM`              | Total sales revenue.                                        |
| `quantity`     | `SUM`              | Total number of units sold.                                 |
| `price`        | Usually not summed | Unit price; commonly analysed using `AVG`, `MIN`, or `MAX`. |
| `order_number` | `COUNT(DISTINCT)`  | Number of unique sales orders.                              |

### Derived Metric

The expected relationship between the main sales measures is:

```text
sales_amount = quantity × price
```

This relationship should be validated during silver-layer data-quality checks.

### Source Mapping

| Gold Column     | Primary Source                                                   |
| --------------- | ---------------------------------------------------------------- |
| `order_number`  | `silver.crm_sales_details.sls_ord_num`                           |
| `product_key`   | Lookup from `gold.dim_products` using the product business key   |
| `customer_key`  | Lookup from `gold.dim_customers` using the customer business key |
| `order_date`    | `silver.crm_sales_details.sls_order_dt`                          |
| `shipping_date` | `silver.crm_sales_details.sls_ship_dt`                           |
| `due_date`      | `silver.crm_sales_details.sls_due_dt`                            |
| `sales_amount`  | `silver.crm_sales_details.sls_sales`                             |
| `quantity`      | `silver.crm_sales_details.sls_quantity`                          |
| `price`         | `silver.crm_sales_details.sls_price`                             |

## Business Rules

### Customer Rules

- Each customer should appear once in `gold.dim_customers`.
- Customer data is enriched using ERP source data.
- Missing or unknown categorical values should be standardised to `n/a`.
- CRM gender values take precedence where CRM and ERP gender values conflict.

### Product Rules

- Each current product should appear once in `gold.dim_products`.
- Historical product records with a populated end date are excluded from the current product dimension.
- Product categories are enriched using ERP category data.
- Missing costs should be standardised according to the silver-layer transformation rules.

### Sales Rules

- Each fact row represents a sales-order line item.
- Every sales row should resolve to a valid customer and product surrogate key.
- `sales_amount` should equal `quantity × price`, unless a documented business rule explains otherwise.
- Shipping and due dates should not normally occur before the order date.
- Sales measures should not contain negative values unless returns or adjustments are explicitly supported.

## Example Analytical Queries

The Gold Layer supports questions such as:

- What is total sales revenue by country?
- Which product categories generate the most revenue?
- How many products were sold each month?
- Who are the highest-value customers?
- What is the average selling price by product line?
- How many unique orders were placed by year?
- Which products have the highest sales quantity?
- What is the average time between order date and shipping date?
